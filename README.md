## 1. 会议纪要生成
**提示词：**

# 角色
你是一位专业的会议记录员。

# 任务
根据以下会议对话，生成一份标准会议纪要。

# 输出格式要求
1.  **会议主题**：根据内容归纳
2.  **参会人员**：从对话中提取人名
3.  **讨论要点**：列出3-5点，使用项目符号
4.  **决议事项**：明确标注
5.  **待办事项**：以表格形式呈现，包含【负责人、任务、截止日期】三列

# 会议对话内容
对话文字：
张敏： 好，人都齐了，我们开始。距离原定3月15日的上线只剩三天。今天必须把阻塞性问题全部闭环。先看测试，上周报的两个P0级Bug现在状态怎么样？
王茜： 第一个是“工单批量导出超过500条时后端超时”，陈昊昨天修了，我今早回归通过。第二个是“前端在Safari浏览器下附件上传按钮无响应”，李芮还在改，说是浏览器兼容性问题。
李芮： 对，那个问题我定位到了，是一个第三方库在Safari下的已知Bug。我换了另一个库，但需要后端配合改一下接口的返回字段，多加一个fileId。改动很小，陈昊你今天能帮忙吗？
陈昊： （叹气）又是改接口？我这周已经加了三次临时字段了。具体要加什么？
李芮： 就是上传成功后，后端原来只返回url，我需要同时返回fileId，否则新库没法绑定文件预览。
陈昊： 行吧，我下午给你加。但今晚我要发一个hotfix，你得在晚上8点前联调完，不然我就封版了。
张敏： 李芮，这个时间你接不接受？
李芮： 可以，我晚上加班等联调。
张敏： 好，这条记录为今天的待办。王茜，其他P1级Bug呢？不影响上线的可以降级处理。
王茜： 还有三个P1：一个是“搜索工单时按时间排序偶尔乱序”，这个复现概率很低，我建议延后到下个迭代。另外两个都是UI问题，不阻塞流程。不过我有个新风险——性能测试昨天跑出来，在并发50人同时导出工单时，数据库CPU会冲到90%持续10秒。赵磊，这个线上能扛住吗？
赵磊： 50人同时导出？实际业务场景不太可能。我们生产环境工单导出功能很少超过5人并发。90% CPU持续10秒不会导致服务中断，我觉得可以先上线观察，后续做读写分离。
张敏： 陈昊你什么意见？毕竟是你写的导出接口。
陈昊： 我同意赵磊。这个导出本来就不是高频操作，为了低频场景做大重构不划算。我可以在接口上加一个限流，每秒最多处理5个导出请求，超过的排队。这个今天也能加上。
张敏： 限流方案可以。那性能问题不作为上线阻塞项。王茜你还有别的吗？
王茜： 暂时没有。对了，回归测试还需要一个干净的测试环境，这周其他项目组一直在占着，赵磊你明天上午能帮我们单独切一套环境吗？只需要用半天。
赵磊： 明天上午10点前我搞定。你提前把需要的配置发我。
王茜： 没问题，会后我发。
张敏： （看手机）还有一个事，运营那边刚才在群里问，新版本上线后旧的工单模板会不会失效？李芮，你负责前端迁移逻辑，确认一下。
李芮： 不会失效，我们有向下兼容。但旧模板里如果使用了自定义字段，用户需要手动刷新一次页面。这个我已经写在发布公告草稿里了。
张敏： 好的，那我把这个告知运营。最后，我们过一下上线时间点。陈昊，你的hotfix和限流今天能搞定吗？
陈昊： 限流和给前端的fileId字段，今天下午6点前都能提测。王茜你晚上跑一下回归。
王茜： 只要环境没问题，我今晚能跑完。
张敏： 那明天上午由赵磊执行预发布环境验证，没问题的话周四下午2点正式上线。大家有没有异议？
众人： （沉默）没有。
张敏： 好。我最后总结一下今天的关键结论和待办，稍后会发到群里。散会。
